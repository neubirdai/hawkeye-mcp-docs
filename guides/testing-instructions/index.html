    Testing Instructions¶

    Learn to test and refine investigation instructions before deploying them.

    Why Test Instructions?¶

    Problem: Bad instructions affect ALL future investigations.

    Solution: Test instructions on past investigations before adding to your project.

    graph LR
    A[Write Instruction] --> B[Validate]
    B --> C[Apply to Test Session]
    C --> D[Rerun Investigation]
    D --> E[Compare RCAs]
    E -->|Better| F[Add to Project]
    E -->|Worse| G[Refine & Retry]

    Quick Start¶

    5-Step Testing Workflow¶

    1. Pick a past investigation as test case
2. Validate your instruction
3. Apply instruction to that session
4. Rerun the investigation
5. Compare new RCA with original

    Only add to project if the new RCA is better!

    Step-by-Step Guide¶

    Step 1: Choose a Test Session¶

    Find a past investigation to test against:

    Show me investigations from the last 7 days

    Pick one that represents the type of incident your instruction targets.

    Example: 

    I want to test an instruction for database issues.
Show me database-related investigations.

    Save the session_uuid from the results.

    Step 2: Write Your Instruction¶

    Draft the instruction content:

    Example - Database Investigation: 

    For database performance issues:
1. Check EXPLAIN ANALYZE output for slow queries
2. Identify missing indexes
3. Review connection pool utilization
4. Check for lock contention
5. Suggest specific index improvements with CREATE INDEX commands

    Instruction Types¶

    FILTER Instructions¶

    Purpose: Reduce noise by filtering out low-value alerts

    Example: 

    Only investigate incidents with:
- Severity P1 or P2
- Affecting production environment
- Not from load testing systems

    Testing: 1. Apply to multiple past sessions 2. Check if important alerts would be filtered 3. Ensure noise is actually reduced

    SYSTEM Instructions¶

    Purpose: Provide architectural context

    Example: 

    Our infrastructure:
- Microservices on AWS EKS (5 services)
- PostgreSQL RDS with read replicas
- Redis ElastiCache for caching
- Datadog APM for distributed tracing
- Peak traffic: 1000 req/sec 2-6pm EST

    Testing: 1. Apply to various incident types 2. Check if context improves root cause analysis 3. Ensure no irrelevant information

    GROUPING Instructions¶

    Purpose: Group related alerts together

    Example: 

    Group incidents when:
- Same service and error type within 15 minutes
- Same root cause (deployment, database, etc.)
- Cascading failures from single source

    Testing: 1. Review past incidents that should be grouped 2. Apply instruction and rerun 3. Verify correct grouping behavior

    RCA Instructions¶

    Purpose: Guide investigation behavior

    Example: 

    For API latency issues:
1. Check Datadog APM traces first
2. Identify slowest span in trace
3. Review database query performance
4. Check external API timeouts
5. Examine cache hit rates
6. Provide specific optimization recommendations

    Testing: 1. Apply to past latency incidents 2. Check if investigation follows steps 3. Verify better remediation suggestions

    Advanced Testing Techniques¶

    A/B Testing Instructions¶

    Test two different approaches:

    Approach A: 

    For deployment failures, check container logs first

    Approach B: 

    For deployment failures, check health check endpoints first

    1. Apply Approach A to session-1, rerun, save RCA
    2. Apply Approach B to session-1, rerun, save RCA
    3. Compare which provides better insights
    4. Use the better one

    Multi-Session Testing¶

    Test instruction across multiple sessions:

    # Test against 5 different database incidents
Apply this instruction to sessions: [uuid1, uuid2, uuid3, uuid4, uuid5]
Rerun all of them
Compare all RCAs

    If ⅘ improved, the instruction is likely good.

    Negative Testing¶

    Test that instructions don't harm unrelated investigations:

    # Instruction for database issues
# Test on API latency incident (unrelated)

    Ensure the instruction doesn't degrade investigations for other incident types.

    Instruction Best Practices¶

    Do's âœ…¶

    • Be specific - "Check PostgreSQL slow queries" vs "Check database"
    • Provide steps - Numbered investigation workflow
    • Include context - Architecture details, business logic
    • Test thoroughly - Multiple sessions, different incident types
    • Iterate - Refine based on results

    Don'ts âŒ¶

    • Don't be vague - "Investigate thoroughly" doesn't help
    • Don't contradict - Ensure instructions don't conflict
    • Don't over-constrain - Allow flexibility in investigation
    • Don't skip testing - Always test before deploying
    • Don't set and forget - Review and update regularly

    Common Testing Scenarios¶

    Scenario 1: Database Query Optimization¶

    Instruction: 

    For slow database queries:
1. Get EXPLAIN ANALYZE output
2. Identify missing indexes
3. Calculate selectivity of proposed indexes
4. Provide CREATE INDEX commands
5. Estimate performance improvement

    Test on: - Slow query incidents (should improve) - Connection pool issues (should not degrade) - Database crashes (should not interfere)

    Scenario 2: Deployment Rollback¶

    Instruction: 

    For incidents following deployments:
1. Identify which service was deployed
2. Get deployment timestamp
3. Correlate incident timing with deployment
4. Provide rollback command if deployment is cause
5. List changed files in deployment

    Test on: - Post-deployment failures (should improve) - Unrelated incidents during deployment window (should not blame deployment) - Successful deployments (should not create false positives)

    Scenario 3: Memory Leak Detection¶

    Instruction: 

    For memory-related incidents:
1. Plot memory growth timeline
2. Identify leak rate (MB/hour)
3. Check for unclosed connections
4. Review cache growth patterns
5. Examine heap dump if available
6. Suggest resource cleanup code

    Test on: - Confirmed memory leaks (should improve) - Legitimate memory growth (traffic increase) (should distinguish) - CPU issues (should not confuse)

    Refining Instructions¶

    If test results are mixed:

    Iteration 1: Too Generic¶

    Original: 

    Check for performance issues

    Problem: Too vague, doesn't guide investigation

    Refined: 

    For performance issues:
1. Identify bottleneck (CPU, memory, I/O, network)
2. Get metrics for bottleneck resource
3. Find peak usage times
4. Correlate with application behavior

    Iteration 2: Too Specific¶

    Original: 

    Always check table users for lock contention

    Problem: Too narrow, not applicable to all DB issues

    Refined: 

    For database lock issues:
1. Identify locked tables from pg_locks
2. Find blocking queries
3. Check lock wait times
4. Suggest query optimization or table partitioning

    Iteration 3: Adding Context¶

    Original: 

    Check API response times

    Enhanced: 

    Check API response times:
- Normal: <100ms (p95)
- Degraded: 100-500ms
- Critical: >500ms
- Check /health endpoint separately (should be <10ms)
