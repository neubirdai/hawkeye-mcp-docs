    Complete Onboarding Guide¶

    Step-by-step guide to set up Hawkeye MCP from scratch to your first investigation.

    Overview¶

    This guide walks you through the complete onboarding process:

    1. ‚úÖ Install and configure Hawkeye MCP
    2. ‚úÖ Create your first project
    3. ‚úÖ Add cloud connections
    4. ‚úÖ Configure investigation instructions
    5. ‚úÖ Run your first investigation
    6. ‚úÖ Optimize and refine

    Time required: 30-45 minutes

    Prerequisites¶

    Before starting, ensure you have:

    • Hawkeye account (book demo if needed)
    • Node.js 20+ installed
    • MCP client installed (Claude Desktop recommended)
    • Cloud provider access (AWS, Azure, or GCP)
    • Monitoring tool credentials (Datadog, PagerDuty, etc.)

    Phase 1: Installation & Setup¶

    Step 1.1: Install Hawkeye MCP¶

    Choose your installation method:

    npx hawkeye-mcp-server

    npm install -g hawkeye-mcp-server
hawkeye-mcp --version

    Step 1.2: Configure Claude Desktop¶

    Edit your Claude Desktop config:

    macOS: 

    nano ~/Library/Application\ Support/Claude/claude_desktop_config.json

    Add configuration: 

    {
  "mcpServers": {
    "hawkeye": {
      "command": "npx",
      "args": ["-y", "hawkeye-mcp-server"],
      "env": {
        "HAWKEYE_EMAIL": "your-email@company.com",
        "HAWKEYE_PASSWORD": "your-password",
        "HAWKEYE_BASE_URL": "https://app.neubird.ai/api"
      }
    }
  }
}

    Step 1.3: Restart and Verify¶

    1. Quit Claude Desktop completely (Cmd+Q on macOS)
    2. Relaunch Claude Desktop
    3. Check tools panel (üî® icon) - should show "hawkeye"

    Test it: 

    List my Hawkeye projects

    Installation Complete

    If you see a list of projects (even if empty), you're ready to proceed!

    Phase 2: Create Your First Project¶

    Step 2.1: Understand Projects¶

    Hawkeye projects organize: - Connections - Which cloud/monitoring tools to use - Instructions - How to investigate incidents - Sessions - Investigation history

    Most teams start with one project per environment (Production, Staging, etc.).

    Step 2.2: Create Project¶

    Ask Claude:

    Create a new Hawkeye project called "Production" for my production environment

    Claude will use hawkeye_create_project and return:

    ‚úì Created project "Production"
UUID: abc-123-def-456
Status: Active

Next steps:
1. Add connections
2. Configure instructions
3. Start investigating

    Save the Project UUID - You'll need it for later steps.

    Step 2.3: Verify Project¶

    Show me details for my Production project

    This uses hawkeye_get_project_details to show: - Project configuration - Connected data sources (currently empty) - Active instructions (currently none)

    Step 2.4: Set as Default Project¶

    Set this project as your default to avoid specifying project_uuid in every command:

    Set Production as my default project

    Or use the UUID directly:

    hawkeye_set_default_project(project_uuid="abc-123-def-456")

    Claude will confirm:

    ‚úì Default project set to: Production
UUID: abc-123-def-456

All operations will now use this project by default.

    Benefits: - üéØ No need to specify project_uuid in commands - üîÑ Easy switching between environments later - üí¨ Use natural language: "Switch to Staging project"

    Multiple Projects

    If you create multiple projects (Production, Staging, Dev), you can quickly switch between them using hawkeye_set_default_project. The default persists for your entire MCP session.

    Phase 3: Add Cloud Connections¶

    Step 3.1: Choose Your Connections¶

    Hawkeye supports:

    Connection Type Use For
    AWS CloudWatch logs/metrics, EC2, RDS, Lambda, ECS
    Azure Azure Monitor, App Insights, VMs, Functions
    GCP Cloud Logging, Cloud Monitoring, Compute, GKE
    Datadog Logs, metrics, traces, APM
    PagerDuty Alert management, on-call schedules
    New Relic APM, infrastructure monitoring

    Start with 1-2 connections

    Begin with your primary cloud provider + main monitoring tool.

    Step 3.2: AWS Connection Setup¶

    Prepare AWS Credentials¶

    You'll need: 1. IAM Role ARN with read-only access 2. External ID for security 3. Regions to monitor

    Create IAM Role:

    # Use AWS CloudFormation or create manually
# Required permissions: ReadOnlyAccess policy
# Trust relationship: Allow Hawkeye to assume role

    Ask Claude for help:

    How do I create an AWS IAM role for Hawkeye?

    This uses hawkeye_get_guidance to provide step-by-step AWS setup instructions.

    Create AWS Connection¶

    Create an AWS connection named "Production AWS" with:
- Role ARN: arn:aws:iam::123456789012:role/HawkeyeReadOnly
- External ID: hawkeye-prod-external-id
- Regions: us-east-1, us-west-2

    Phase 4: Configure Instructions¶

    Step 4.1: Understand Instruction Types¶

    Type Purpose Example
    FILTER Reduce noise "Only investigate P1 and P2 incidents"
    SYSTEM Investigation context "Our architecture uses microservices on EKS"
    GROUPING Group related alerts "Group alerts from same service"
    RCA Investigation steps "For database issues, check slow query logs first"

    Step 4.2: Create System Instruction¶

    Provide context about your infrastructure:

    Create a SYSTEM instruction for my Production project:

"Our production environment runs on AWS EKS with:
- 5 microservices: api-gateway, user-service, order-service,
  payment-service, notification-service
- PostgreSQL RDS for data persistence
- Redis ElastiCache for caching
- All services instrumented with Datadog APM
- Logging centralized in CloudWatch and Datadog
- Peak traffic: 1000 req/sec from 2pm-6pm EST"

    This uses hawkeye_create_project_instruction:

    ‚úì Created SYSTEM instruction
Name: "Production Architecture Context"
Status: Active

This will be included in all investigations to provide
architectural context.

    Step 4.3: Create Filter Instruction¶

    Reduce noise from low-priority alerts:

    Create a FILTER instruction for my Production project:

"Only investigate incidents with severity P1 or P2.
Ignore P3, P4, and informational alerts."

    ‚úì Created FILTER instruction
Name: "Priority Filter"
Status: Active

This will filter out P3+ alerts automatically.

2. Look for database slow queries
3. Check Redis cache hit rate
4. Review EKS pod resource utilization
5. Check for external API timeouts"

    ‚úì Created RCA instruction

    Step 5.5: Ask Follow-Up Questions¶

    Why wasn't this caught in staging?

    This uses hawkeye_continue_investigation:

    Analyzing staging environment...

The issue wasn't caught in staging because:

1. Staging database has only 1,000 user records
   Production has 500,000 records

2. Without index, query time:
   - Staging: ~5ms (acceptable)
   - Production: ~800ms (unacceptable)

3. Staging load tests run at 50 req/sec
   Production peak is 200 req/sec

4. The performance degradation only becomes apparent
   at production scale

RECOMMENDATION:
Add production-scale load testing:
- Use anonymized production data snapshots
- Run load tests at 200+ req/sec
- Monitor database query times during tests

No alerts fired yet but trending upward.

    Claude will use hawkeye_create_manual_investigation:

    ‚úì Created manual investigation

    Wait for completion:

    Show me the RCA for this investigation

    Best practices for manual investigations:

    ‚úÖ Good prompt: 

    Investigate database connection pool issues in postgres-primary.
Between 2pm-3pm EST yesterday. Connection count hit max of 100.

    ‚ùå Bad prompt: 

    Something seems wrong with the database

    Tip: Include service names, timeframes, and specific symptoms for best results. If your prompt is too vague, Claude will ask for clarification.
